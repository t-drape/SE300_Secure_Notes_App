1. Introduction and Purpose:

The System Under Development (SUD) is a local, command-line-based secure notes application designed to provide users with a privacy-focused method for creating, managing, and analyzing personal notes. The SUD enables users to create new notes, read previously saved notes, append additional content to existing notes, delete existing notes, and generate AI-based summaries and keyword analyses of selected notes. The primary purpose of the SUD is to ensure that all user-generated note content remains confidential and protected against unauthorized access. To achieve this, all notes are encrypted using AES-256 encryption before being written to the host computer’s storage. Encryption keys are derived from a user-provided password using the PBKDF2 (Password-Based Key Derivation Function 2) algorithm. Decryption occurs only in memory when explicitly requested by the user, and plaintext content is never stored persistently on disk. The SUD operates entirely offline and requires no external servers, databases, or cloud services. All functionality is executed locally on the host computer (HC) through a menu-driven command-line interface (CLI). The system is designed to be minimalistic, secure, and modular, with a clear separation of concerns between file management, encryption, authentication, and AI-based analysis components.

2 Design Considerations:

The design of the SUD is primarily shaped by its security and privacy requirements. Since the application handles personal notes, protecting user data is the most important consideration. All notes must remain encrypted when stored on the Host Computer.

Another key consideration is that the SUD operates entirely offline using a command-line interface. This means all functionality must run locally without relying on external servers, databases, or network connections. The design must therefore securely manage data persistence using local files while maintaining a clear separation between encrypted storage and runtime operations.

Modularity is also a major focus. The system is divided into separate components for file handling, encryption, AI processing, and user interaction. This separation keeps responsibilities clear, reduces complexity, and makes the system easier to test and maintain.

Performance and reliability are additional considerations. Encryption, decryption, and summarization operations must meet the time requirements defined in the SRD. The design must also avoid uncontrolled behavior, such as infinite loops or unsafe state transitions, to ensure the system can be verified and tested effectively.

2.1 Assumptions and Dependencies:

The SUD is designed to operate on a Host Computer with Python 3.10 or later installed. It assumes the user has access to a functioning command-line interface and can execute Python scripts. The system relies on the local file system to store encrypted note files and therefore requires read and write permissions within the project directory.

The design assumes compatibility with common operating systems that support Python and standard file system operations, including Windows, macOS, and Linux. No operating system–specific features are required, and the SUD is intended to remain platform-independent.

The SUD does not require internet connectivity, external servers, or database systems. All functionality is expected to run locally without reliance on network services.

The design also anticipates possible future enhancements, such as additional AI functionality, alternative user interfaces (e.g., graphical interface), or expanded note management features. The system's modular structure is intended to support such extensions without requiring major architectural changes.

2.3 Goals and Guidelines:

The SUD has been designed with modifiability, maintainability, security, and performance in mind (see Section 2.1 for potential changes). Consequently, sound software design principles have been emphasized, especially:
    - Modularity and information hiding: The design separates concerns to enable independent modification and testing of encryption, note management, user interface, and AI processing              functionality.
    - Loose coupling between modules: Dependencies are minimized to allow changes without cascading effects throughout the system.
    - High cohesion within modules: Each module has a single, well-defined responsibility.
    - Partitioning volatility: Components likely to change are isolated from stable components to accommodate future requirements with minimal impact.
    - Security focused design: The architecture enforces AES-256 encryption, PBKDF2 key derivation, and ensures plaintext content never persists outside application memory.
